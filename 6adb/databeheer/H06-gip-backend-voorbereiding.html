GIP Backend voorbereiding

verdieping 4 u

🎯 Leerdoelen
  • De leerling kan een genormaliseerd databaseschema ontwerpen voor het GIP-project.
  • De leerling kan API-endpoints documenteren vóór de implementatie (HTTP-methode, pad, body, response).
  • De leerling kan de samenwerking tussen front-end, back-end en database beschrijven en implementeren.
  • De leerling kan een API testen met Postman of curl vóór de front-end gekoppeld wordt.

1. Databaseschema ontwerpen voor het GIP

Vóór je aan de code begint, ontwerp je het databaseschema. Een goed schema is de basis van een werkende applicatie.

Stappenplan
  1. Bepaal de entiteiten (objecten in je applicatie, bv. Gebruiker, Product, Bestelling)
  2. Bepaal de attributen (kolommen) per entiteit
  3. Bepaal de relaties (één-op-veel, veel-op-veel)
  4. Teken een ERD (Entity Relationship Diagram)
  5. Normaliseer het schema tot minstens 3NF
  6. Schrijf de CREATE TABLE statements

Voorbeeld: schema voor een eenvoudige takenapp (GIP-voorbeeld):

-- schema.sql
CREATE DATABASE IF NOT EXISTS takenapp_db;
USE takenapp_db;

CREATE TABLE gebruikers (
  gebruiker_id INT AUTO_INCREMENT PRIMARY KEY,
  gebruikersnaam VARCHAR(50)  NOT NULL UNIQUE,
  email          VARCHAR(150) NOT NULL UNIQUE,
  wachtwoord     VARCHAR(255) NOT NULL,  -- gehashed opslaan!
  aangemaakt_op  DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE categorieën (
  categorie_id INT AUTO_INCREMENT PRIMARY KEY,
  naam         VARCHAR(100) NOT NULL,
  kleur        CHAR(7) DEFAULT '#6C63FF'  -- CSS-kleurcode
);

CREATE TABLE taken (
  taak_id      INT AUTO_INCREMENT PRIMARY KEY,
  gebruiker_id INT  NOT NULL,
  categorie_id INT,
  titel        VARCHAR(200) NOT NULL,
  beschrijving TEXT,
  gedaan       BOOLEAN DEFAULT FALSE,
  vervaldatum  DATE,
  prioriteit   ENUM('laag','normaal','hoog') DEFAULT 'normaal',
  aangemaakt_op DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (gebruiker_id) REFERENCES gebruikers(gebruiker_id) ON DELETE CASCADE,
  FOREIGN KEY (categorie_id) REFERENCES categorieën(categorie_id) ON DELETE SET NULL
);

3. Koppeling front-end ↔ back-end ↔ database

Onderstaand schema toont hoe de drie lagen samenwerken voor één gebruikerssactie ("alle taken laden"):

// FRONT-END (browser — app.js)
async function laadTaken() {
  const response = await fetch("/api/taken");
  const taken    = await response.json();
  toonTakenInDOM(taken);
}

// ↕ HTTP GET /api/taken

// BACK-END (Express — routes/taken.js)
router.get("/", async (req, res) => {
  try {
    const [rijen] = await pool.query(
      "SELECT t.*, c.naam AS categorie_naam FROM taken t LEFT JOIN categorieën c ON t.categorie_id = c.categorie_id WHERE t.gebruiker_id = ?",
      [req.user.id]
    );
    res.json(rijen);
  } catch (fout) {
    res.status(500).json({ fout: "Kon taken niet laden" });
  }
});

4. Testen met Postman en curl

Test je API grondig vóór je de front-end koppelt. Gebruik Postman (grafische interface) of curl (terminal).

# curl voorbeelden
# GET alle taken
curl http://localhost:3000/api/taken

# POST nieuwe taak
curl -X POST http://localhost:3000/api/taken \
  -H "Content-Type: application/json" \
  -d '{"titel":"Studeren voor databeheer","prioriteit":"hoog"}'

# PATCH taak als gedaan
curl -X PATCH http://localhost:3000/api/taken/1/gedaan

# DELETE taak
curl -X DELETE http://localhost:3000/api/taken/1
Postman

Postman is een gratis tool waarmee je HTTP-aanvragen grafisch kan opbouwen en uitvoeren. Je kan er ook testcollecties mee opbouwen en automatisch testen schrijven die de responsedok controleren. Ideaal voor het testen van je GIP-API.

Samenvatting

Samenvatting
  • Start het GIP altijd met een ERD en genormaliseerd schema (tot 3NF).
  • Documenteer alle API-endpoints vóór je begint met coderen.
  • Test elke route grondig met Postman of curl vóór je de front-end koppelt.
  • Gebruik geparametriseerde queries, valideer input en geef correcte statuscodes.
  • De drie lagen (front-end, back-end, database) communiceren via HTTP en SQL.
