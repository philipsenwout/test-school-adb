Normalisatie

verdieping 4 u
[LD-DB-09] Databaseschema normaliseren tot 3NF — verdieping

1. Waarom normaliseren?

Een slecht ontworpen database leidt tot drie typische problemen:

ProbleemOmschrijvingVoorbeeld
Invoeg-anomalieJe kan een feit niet opslaan zonder ander feitKlant toevoegen zonder bestelling
Bijwerk-anomalieDezelfde info staat op meerdere plaatsen → inconsistentieAdres klant op 10 rijen aanpassen
Verwijder-anomalieVerwijderen van iets wist ook andere infoLaatste bestelling verwijderen wist klantengegevens

Normalisatie is het proces waarbij je een databaseschema stap voor stap verbetert volgens vaste regels (normaalvormen) om deze anomalieën te vermijden.

2. Eerste normaalvorm (1NF)

Definitie — 1NF

Een tabel is in 1NF als:

  • Elke cel bevat slechts één waarde (atomaire waarden)
  • Er zijn geen herhalende groepen of arrays in een cel
  • Elke rij is uniek (er is een primaire sleutel)

Niet in 1NF (probleem: meerdere telefoonnummers in één cel):

klant_idnaamtelefoon
1Jan Peeters0472 11 22 33, 0498 44 55 66
2Marie Janssen0456 77 88 99

In 1NF (telefoonnummers in aparte tabel):

2Marie Janssen

3. Tweede normaalvorm (2NF)

Definitie — 2NF

Een tabel is in 2NF als:

  • Ze in 1NF is
  • Elk niet-sleutelattribuut volledig functioneel afhankelijk is van de volledige primaire sleutel (geen partiële afhankelijkheid)

2NF is enkel relevant bij samengestelde primaire sleutels.

Niet in 2NF (primaire sleutel = bestelling_id + product_id, maar product_naam hangt enkel af van product_id):

In 2NF (product_naam verhuist naar aparte producten-tabel):

4. Derde normaalvorm (3NF)

Definitie — 3NF

Een tabel is in 3NF als:

  • Ze in 2NF is
  • Er zijn geen transitieve afhankelijkheden: een niet-sleutelattribuut mag niet afhangen van een ander niet-sleutelattribuut

Niet in 3NF (stad hangt af van postcode, niet van klant_id):

klant_idnaampostcodestad
1Jan Peeters2000Antwerpen
2Marie Janssen9000Gent
3Lien Claes2000Antwerpen

In 3NF (postcode-stad naar aparte tabel):

klant_idnaampostcode
1Jan Peeters2000
2Marie Janssen9000
postcodestad
2000Antwerpen
9000Gent

5. Volledig voorbeeld: stap voor stap

We beginnen met deze niet-genormaliseerde tabel voor een boekhandel:

bestelling_idklant_naamklant_emailklant_stadpostcodeboeken (titels)boek_auteurs
1Jan Peetersjan@mail.beAntwerpen2000Clean Code, RefactoringR. Martin, M. Fowler
2Marie Janssenmarie@mail.beGent9000Clean CodeR. Martin

Na normalisatie tot 3NF:

-- 3NF schema
CREATE TABLE steden (
  postcode CHAR(4) PRIMARY KEY,
  stad     VARCHAR(100)
);

CREATE TABLE klanten (
  klant_id  INT AUTO_INCREMENT PRIMARY KEY,
  naam      VARCHAR(100),
  email     VARCHAR(150) UNIQUE,
  postcode  CHAR(4),
  FOREIGN KEY (postcode) REFERENCES steden(postcode)
);

CREATE TABLE boeken (
  boek_id INT AUTO_INCREMENT PRIMARY KEY,
  titel   VARCHAR(200),
  auteur  VARCHAR(100)
);

CREATE TABLE bestellingen (
  bestelling_id INT AUTO_INCREMENT PRIMARY KEY,
  klant_id      INT,
  besteldatum   DATE,
  FOREIGN KEY (klant_id) REFERENCES klanten(klant_id)
);

CREATE TABLE bestelling_boeken (
  bestelling_id INT,
  boek_id       INT,
  aantal        INT DEFAULT 1,
  PRIMARY KEY (bestelling_id, boek_id),
  FOREIGN KEY (bestelling_id) REFERENCES bestellingen(bestelling_id),
  FOREIGN KEY (boek_id)       REFERENCES boeken(boek_id)
);

Samenvatting

Samenvatting
  • Slechte schemas leiden tot invoeg-, bijwerk- en verwijderanomalieën.
  • 1NF: atomaire waarden, geen herhalende groepen, unieke rijen.
  • 2NF: geen partiële afhankelijkheden van de samengestelde primaire sleutel.
  • 3NF: geen transitieve afhankelijkheden (A → B → C, terwijl A primaire sleutel is).
  • Normaliseren leidt tot meer tabellen maar vermijdt redundantie en anomalieën.

Huistaak H01

Ontwerp een genormaliseerd schema (tot 3NF) voor een schoolbibliotheek:

  1. Leerlingen kunnen boeken uitlenen (meerdere tegelijk).
  2. Boeken hebben een titel, auteur, ISBN en categorie.
  3. Een categorie heeft een naam en een beschrijving.
  4. Bijhouden wanneer een boek werd uitgeleend en wanneer het teruggegeven moet worden.
  5. Lever: ERD-diagram (op papier of via draw.io) + CREATE TABLE SQL-statements.
