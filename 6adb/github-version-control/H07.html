GitHub Issues & Samenwerken

ADB28De leerlingen communiceren over vragen, problemen en updates.
ADB07De leerlingen houden hun administratie bij.

🎯 Leerdoelen
  • Je kan een GitHub Issue aanmaken voor een bug of feature request
  • Je kan labels, assignees en milestones gebruiken om issues te organiseren
  • Je kan een Issue linken aan een commit via fixes #nummer
  • Je kan professioneel communiceren in een Issue (beschrijving, reactie, sluiten)

1. Wat is een Issue?

Een GitHub Issue is een gestructureerde manier om samen te werken aan een project. Je gebruikt Issues om:

  • Bugs te melden — iets werkt niet zoals verwacht
  • Features voor te stellen — iets wat er nog niet is
  • Vragen te stellen aan medeontwikkelaars
  • Taken bij te houden in een project

Issues zijn zichtbaar voor iedereen in het team. Ze fungeren als het "to-do-systeem" van je repository. Elk Issue krijgt automatisch een uniek nummer (bv. #1, #2, #3, …).

📌 Waar vind je Issues?

Je GitHub-repository heeft een tabblad Issues. Klik erop om alle openstaande en gesloten issues te zien. Openstaande issues zijn taken die nog gedaan moeten worden — gesloten issues zijn afgewerkt.

2. Een Issue aanmaken

Een goed gestructureerd Issue maakt het voor iedereen duidelijk wat het probleem is en hoe het opgelost kan worden. Volg deze stappen:

  1. Ga naar je repository op GitHub
  2. Klik op het tabblad Issues
  3. Klik op New issue
  4. Geef een duidelijke titel (bv. "Knop 'Opslaan' reageert niet op mobiel")
  5. Schrijf een beschrijving: wat is het probleem? Hoe reproduceer je het? Wat verwacht je?
  6. Klik op Submit new issue

Voorbeeld van een goede issue-beschrijving

Gebruik Markdown-opmaak om je issue leesbaar te maken. Hieronder zie je een template die je kan hergebruiken:

Markdown
## Beschrijving
De knop "Opslaan" op de formulierpagina reageert niet wanneer je
op een mobiel scherm (< 480px) bent.

## Stappen om te reproduceren
1. Open de pagina op een mobiel scherm of gebruik DevTools (mobiele emulatie)
2. Vul het formulier in
3. Klik op "Opslaan"
4. → Niets gebeurt

## Verwacht gedrag
Het formulier wordt opgeslagen en de gebruiker ziet een bevestigingsbericht.

## Schermafbeelding
[Voeg hier een screenshot toe]

## Omgeving
- Browser: Chrome 124
- Schermgrootte: 375 × 812 px
⚠ Wees specifiek

Een vage issue zoals "het werkt niet" is niet nuttig. Wees altijd specifiek: wat deed je, wat verwachtte je, wat gebeurde er werkelijk? Hoe gedetailleerder je Issue, hoe sneller iemand het kan oplossen.

3. Labels & Assignees

GitHub biedt extra tools om issues te organiseren. De drie belangrijkste zijn labels, assignees en milestones.

Labels

Labels helpen issues te categoriseren zodat je snel de juiste issues vindt. GitHub maakt standaard een aantal labels aan:

Label Betekenis
bug Iets werkt niet correct
enhancement Nieuwe functionaliteit of verbetering
question Een vraag aan het team
good first issue Geschikt voor beginners
help wanted Extra hulp gevraagd
duplicate Dit issue bestaat al
wontfix Dit wordt niet opgelost

Assignees

Assignees zijn de personen die verantwoordelijk zijn voor het oplossen van de issue. Je kan jezelf of een teamgenoot aanwijzen. Zo is het voor iedereen duidelijk wie waarvoor verantwoordelijk is.

Milestones

Milestones groeperen issues onder een deadline of doel. Voorbeelden:

  • "Sprint 1" — alles wat af moet zijn voor week 4
  • "GIP indiening week 10" — alle taken die nodig zijn voor de GIP-indiening
  • "Versie 1.0" — alles wat in de eerste versie moet zitten

4. Issue linken aan een commit

Je kan een Issue automatisch sluiten via een commit-boodschap. Dit werkt door een speciaal sleutelwoord gevolgd door het Issue-nummer te gebruiken:

Bash
git commit -m "fixes #3: knop reageert nu ook op mobiel"

Wanneer deze commit gepusht wordt naar de main branch, sluit GitHub Issue #3 automatisch. Je hoeft het Issue dus niet handmatig te sluiten.

Sleutelwoorden die GitHub herkent

GitHub herkent de volgende woorden (en varianten) om een Issue te sluiten:

fixes / fix
closes / close
resolves / resolve
📌 Issue-nummer vinden

Het issue-nummer vind je in de URL van het issue (bv. github.com/jou/repo/issues/3 → nummer is 3) of naast de titel in de Issues-lijst.

5. Professioneel communiceren

GitHub Issues zijn openbare communicatiekanalen. In open-source projecten en professionele teams lezen collega's, opdrachtgevers en soms de hele wereld mee. Professioneel schrijven is een vaardigheid die je in je latere loopbaan dagelijks zal gebruiken.

Tips voor professionele communicatie

  • Reageer altijd beleefd en constructief, ook als iemand een fout maakte
  • Gebruik @naam om iemand specifiek te taggen in een reactie (bv. @jantje Kan jij dit nakijken?)
  • Gebruik > citaat om te citeren waarop je reageert
  • Markeer een Issue als closed wanneer het opgelost is — met een korte uitleg wat er veranderd is
  • Gebruik emoticons (👍, ✅) spaarzaam — enkel als ze iets toevoegen
  • Vermijd afkortingen die niet iedereen begrijpt
💼 Professioneel schrijven

In open-source projecten communiceert iedereen via Issues en Pull Requests. Wat je schrijft in een Issue is zichtbaar voor iedereen die je repository bekijkt — nu en in de toekomst. Schrijf altijd alsof je leerkracht of een toekomstige werkgever meeleest.

⚠ Issues zijn geen chat

Gebruik Issues niet als chatruimte voor korte berichten. Schrijf duidelijke, volledige berichten. Als je snel iets wil vragen aan een teamgenoot, gebruik dan een andere tool (bv. Teams). Gebruik Issues voor gestructureerde communicatie over taken en bugs.

Oefeningen

ADB28 Oefening 1 — Issue aanmaken voor een bug

Maak een Issue aan in de GitHub-repository van je GitHub-workflow-oefening (H06). Beschrijf een fictieve bug in de opmaak van je pagina. Gebruik de template uit sectie 2 als basis.

  1. Ga naar je repository op GitHub
  2. Klik op het tabblad Issues → New issue
  3. Geef een duidelijke titel (bv. "Navigatiebalk toont niet correct op mobiel")
  4. Vul de beschrijving in met: stappen om te reproduceren, verwacht gedrag, werkelijk gedrag
  5. Voeg minstens 1 label toe (bv. bug)
  6. Wijs het Issue toe aan jezelf (Assignees)
  7. Dien het Issue in

Controleer: zie je het Issue staan in de Issues-lijst met het juiste label?

ADB16 / ADB28 Oefening 2 — Issue sluiten via commit

Los de fictieve bug uit Oefening 1 op via een commit die het Issue automatisch sluit.

  1. Maak een kleine CSS-aanpassing in je project (bv. voeg flex-wrap: wrap; toe aan de navigatie)
  2. Sla op
  3. Voer uit:
    Bash
    git add .
git commit -m "fixes #1: navigatie toont nu correct op mobiel"
git push
  4. Ga naar GitHub → Issues
  5. Controleer: staat Issue #1 nu bij Closed?

📋 Huistaak — GitHub Issues in jouw GIP-repository

Opdracht: Issues aanmaken en linken in je GIP-repository

Deel 1 — Issue aanmaken (3 punten)

Maak 2 echte Issues aan in jouw GIP-repository (maak die repository alvast aan als je hem nog niet hebt):

  • Issue 1: een feature die je wil bouwen (bv. "Activiteiten filteren op datum")
  • Issue 2: een vraag aan de leerkracht/opdrachtgever (bv. "Moet de admin ook activiteiten van anderen kunnen aanpassen?")

Gebruik labels en schrijf een duidelijke beschrijving voor elk Issue.

Deel 2 — Commit linken (4 punten)

  1. Maak een kleine aanpassing aan je project (bv. README bijwerken)
  2. Commit met fixes #1 of closes #2
  3. Push naar GitHub
  4. Voeg een screenshot toe van het gesloten Issue in je indiening

Deel 3 — Reflectie (3 punten)

Schrijf in 5 zinnen: waarom is het bijhouden van Issues nuttig in een project? Geef een concreet voorbeeld uit je eigen GIP.
